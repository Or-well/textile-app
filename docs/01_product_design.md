# 01 产品设计说明

## 1. 项目定位

本项目是一个面向业余汉化组的本地优先、离线可用、低维护成本的汉化协作工具。

它不是在线翻译平台，也不是通用文本编辑器，而是一个专门用于汉化项目的本地词条翻译工作台。

核心目标：

- 本地运行；
- 离线可用；
- 项目数据保存在本地文件夹；
- 普通成员低门槛参与；
- 支持翻译、校对、审核、术语、任务、评论、统计、导出；
- Git 可作为底层协作与版本追溯机制，但不直接暴露给普通成员；
- 代码和文档保持简约，方便人工维护和 vibe coding。

推荐产品形态：

```text
PWA / 本地网页应用
  ↓
选择本地项目文件夹
  ↓
读取 JSON / JSONL 项目数据
  ↓
离线翻译、校对、审核、统计、导出
  ↓
通过修改包或前端封装 Git 协作
```

## 2. 最终成品应是什么样

最终成品应是一个本地汉化项目工作台。

普通成员看到的是：

```text
打开项目
查看我的任务
翻译 / 校对 / 审核词条
保存
提交任务
同步项目或导出修改包
```

负责人看到的是：

```text
管理项目
分配任务
导入修改
处理冲突
查看统计
导出成品
维护版本
```

技术负责人看到的是：

```text
维护项目格式
维护解析器和导出器
检查数据
处理复杂冲突
使用 Git 追溯和回滚
```

## 4. 用户角色与兼任

成员必须支持多角色兼任。

示例：

```json
{
  "id": "user_b",
  "name": "小B",
  "roles": ["translator", "proofreader", "term_manager"],
  "active": true
}
```

权限计算方式：

```text
最终权限 = 所有角色权限的并集 + 单独允许权限 - 单独禁止权限
```

系统不应要求用户在全局切换唯一身份，而应根据当前页面、当前任务和当前操作自动判断可用功能。

### 4.1 常见角色

```text
owner           项目负责人
admin           管理员
translator      翻译
proofreader     校对
reviewer        审核
term_manager    术语负责人
publisher       发布负责人
tech_lead       技术负责人
readonly        只读成员
```

### 4.2 角色职责

| 角色 | 主要职责 |
|---|---|
| owner | 管理项目、成员、任务、导入修改、处理冲突 |
| admin | 协助管理项目和任务 |
| translator | 翻译词条、评论、提交任务 |
| proofreader | 校对译文、修改译文、标记争议 |
| reviewer | 审核词条、退回词条、处理争议 |
| term_manager | 维护术语表 |
| publisher | 导出成品和报告 |
| tech_lead | 维护格式、解析器、导出器、Git 版本 |
| readonly | 查看项目，不修改 |

## 5. 权限性质

本工具的权限是本地软权限。

它用于：

- 控制界面显示；
- 防止普通成员误操作；
- 规范流程；
- 导入修改包时提示负责人；
- 辅助统计责任归属。

它不能防止：

- 用户手动修改本地文件；
- 用户伪造修改包；
- 用户绕过软件编辑 JSON；
- 恶意用户破坏数据。

设计假设：

```text
成员总体可信。
权限主要防误操作，不防恶意攻击。
负责人维护最终可信主项目。
Git 负责最终追溯。
```

## 6. 任务管理

任务和词条状态必须分开。

词条状态表示词条本身完成度：

```text
untranslated    未翻译
translated      已翻译
proofread       已校对
reviewed        已审核
disputed        有争议
```

任务状态表示工作安排进度：

```text
unassigned      未领取
assigned        已分配
in_progress     进行中
submitted       已提交
completed       已完成
reclaimed       已回收
blocked         有问题
```

任务类型：

```text
translate       翻译
proofread       校对
review          审核
term            术语
export          导出
custom          自定义
```

任务可以按以下方式指定范围：

- 文件；
- 词条范围；
- 词条 ID 列表；
- 任务类型；
- 负责人；
- 截止时间；
- 提交方式。

任务提交方式：

```text
change_package  导出修改包
git_hidden      通过前端封装的同步功能提交
git_manual      由负责人或技术成员手动处理 Git
```

普通成员界面应显示为：

```text
导出修改包
同步提交
由负责人处理
```

## 7. 协作模式

最终产品应支持三种协作模式。

### 7.1 前端封装 Git 协作

Git 作为底层机制，但不直接暴露。

普通成员看到：

```text
同步项目
保存修改
提交任务
上传修改
处理内容冲突
```

底层可能执行：

```text
pull
commit
push
branch
merge
```

但这些术语不出现在普通成员界面。

适合：

- 长期成员；
- 校对；
- 审核；
- 核心翻译；
- 技术能力稍强的成员。

### 7.2 修改包协作

普通成员本地工作后导出 zip 修改包，由负责人导入。

适合：

- 临时成员；
- 完全不想接触同步功能的成员；
- 网络条件差的成员；
- Git 同步失败后的兜底处理。

### 7.3 混合协作

推荐默认模式：

```text
核心成员使用前端封装 Git
临时成员使用修改包
负责人统一维护主项目
技术成员处理复杂 Git 问题
```

## 8. Git 封装原则

普通成员不应直接看到：

```text
clone
pull
push
commit
branch
merge
rebase
remote
origin
HEAD
SHA
checkout
stash
merge conflict
```

应显示为：

| 前端显示 | 底层 Git 行为 |
|---|---|
| 下载项目 | clone |
| 同步项目 | pull / fetch |
| 保存一个版本 | commit |
| 上传修改 | push |
| 我的工作版本 | branch |
| 内容冲突 | merge conflict |
| 版本历史 | log |

Git 错误也应转译：

| 技术错误 | 用户提示 |
|---|---|
| non-fast-forward | 项目有新内容，请先同步 |
| merge conflict | 发现内容冲突，需要处理 |
| authentication failed | 无法连接项目，请检查权限 |
| working tree dirty | 你还有未提交的修改 |

## 9. 简约设计原则

本产品不追求功能堆叠，而追求核心流程稳定。

核心流程：

```text
打开项目
  ↓
查看任务
  ↓
翻译 / 校对 / 审核词条
  ↓
保存修改
  ↓
提交修改
  ↓
负责人合并
  ↓
导出成品
```

默认不做或谨慎做：

```text
实时多人在线编辑
强权限企业系统
复杂审批流
完整 Git 客户端
插件市场
多组织管理
富文本评论
内置 AI 翻译平台
```
