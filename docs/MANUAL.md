# Textile 用户手册

本手册面向普通成员、项目负责人、翻译、校对、审核、术语负责人和发布负责人。内容以当前代码为准；不同成员实际看到的按钮取决于项目角色和个人权限。

## 1. 软件概念

Textile 是本地优先的汉化协作工具：

- 项目数据保存在本地项目文件夹中，也可以导出为 `.hproj` 项目包用于备份、迁移和分发。
- 软件不依赖业务服务器，项目可以离线打开和编辑。
- 成员通过项目内账号登录，不是云账号。
- 普通成员通过修改包提交工作。
- 负责人导入成员修改包、处理冲突，再发布签名项目更新包。
- Git 不是项目协作入口；程序版本发布和项目数据协作是两件事。

Textile 的主要数据包括源文件、词条、术语、任务、评论、成员、权限、日志和项目设置。

## 2. 推荐快速流程

负责人第一次建立项目：

1. 在项目启动中心选择“创建项目”。
2. 选择创建位置，填写项目名称、语言、工作流、进度权重和 owner 账号。
3. 在“文件”页添加源文件。
4. 在“设置”中建立成员、分配角色并确认工作流。
5. 需要签名协作时，每位成员在“设置 > 我的身份密钥”生成并妥善备份自己的身份密钥。
6. 负责人分配任务，成员开始翻译。
7. 成员定期导出普通修改包；负责人导入并处理冲突。
8. 负责人发布签名项目更新包；成员接收更新。
9. 定期导出 `.hproj` 备份，发布前导出成品。

普通成员日常流程：

1. 打开自己的项目副本并登录。
2. 先接收负责人最新的签名项目更新包。
3. 在词条、任务、术语、上下文和评论中工作。
4. 导出“我的全部修改”或任务修改包。
5. 把修改包交给负责人。
6. 负责人发布新项目更新包后，再接收下一轮更新。

接收项目更新前，Textile 会检查当前成员是否有尚未导出的个人修改。若有，必须先导出自己的修改包。

## 3. 项目启动中心

进入位置：打开 Textile 后默认进入 `/projects`。

可以执行：

- 打开 Textile 项目文件夹。
- 导入 `.hproj` 项目文件。
- 预览 `.hproj` 项目文件。
- 创建新项目。
- 从最近项目继续工作。
- 从最近项目列表移除记录。
- 重新进入本次运行中已经打开的项目。

注意事项：

- 最近项目最多保留 12 条。
- 普通文件夹的授权句柄会保存在浏览器 IndexedDB 中，但浏览器可能要求重新授权。
- 预览 `.hproj` 只读取摘要，不写入本地项目。
- 导入 `.hproj` 会要求选择目标位置，并生成新的本地项目文件夹。
- 从最近项目列表移除只删除启动页记录和本地授权记录，不删除磁盘项目数据。

## 4. 创建项目

进入位置：项目启动中心 > “创建项目”。

需要填写：

- 项目名称和简介。
- 源语言和目标语言。
- 是否启用任务、校对和审核。
- 需要的校对次数，范围是 0 到 3。
- 翻译、校对、审核进度权重，合计必须为 100。
- owner 成员名和密码。

创建结果：

- 在所选文件夹中创建 Textile 项目结构。
- 创建唯一 owner 成员。
- 自动登录 owner 并进入文件页。

注意事项：

- 应选择空文件夹。当前代码会阻止覆盖已存在的 Textile 关键数据路径，但不要把它当作任意文件夹清理工具。
- 密码不会以明文保存，而是保存 PBKDF2 哈希和盐。
- owner 密码丢失后不能从哈希还原，应由仍有权限的管理员重置，或从备份恢复。

## 5. 打开本地项目

进入位置：项目启动中心 > “打开 Textile 项目文件夹”。

操作方式：

1. 选择包含 `project.json` 的项目根目录。
2. Textile 校验项目配置、成员、词条、术语和任务等必要路径。
3. 校验成功后进入项目登录页。

注意事项：

- Web 版依赖 File System Access API，建议使用 Chrome 或 Edge。
- 如果浏览器丢失文件夹授权，需要重新选择项目文件夹。
- 项目配置中的每个文件必须有对应的 `entries/<file_id>` 目录，否则项目可能无法打开。
- 源文件目录不是当前打开校验的强制项，但缺失会影响备份完整性和后续源文件操作。

## 6. 导入和预览 `.hproj` 项目包

进入位置：项目启动中心 > “导入 Textile 项目文件”。

`.hproj` 可用于：

- 完整项目备份。
- 在另一台设备迁移项目。
- 发送一个可导入为本地项目的项目副本。

导入方式：

1. 选择 `.hproj` 文件。
2. 选择一个导入位置。
3. Textile 在该位置创建新的本地项目文件夹。
4. 导入成功后进入普通项目登录页，后续编辑写入这个本地项目文件夹。

预览方式：

- “预览 Textile 项目文件”只读取项目名称、项目简介、项目修订、更新时间、内容规模、成员数、语言方向和导入状态。
- 项目简介默认只显示前两行，较长时可以展开查看。
- 预览会显示“导入后会创建文件夹”，这是选择目标位置后将要创建的子文件夹名称。
- 预览不会创建目录，也不会写入任何项目数据。
- 预览面板可以继续执行“导入为本地项目”。

注意事项：

- 导入会保留原 `project_id`，以便修改包和项目更新包继续匹配。
- `.hproj` 不会挂载为可原地修改的真实文件。
- 如果目标位置已有同名导入目录，Textile 会阻止导入，避免覆盖已有项目。
- 完成本地编辑后，如需继续分发，请重新导出 `.hproj` 项目备份。

## 7. 登录与会话恢复

进入位置：打开项目后自动显示项目登录页。

登录信息：

- 成员名必须与 `members.json` 中的有效成员匹配。
- 密码在本机通过 PBKDF2 校验。
- 停用成员不能恢复会话或登录。

会话恢复：

- 登录后默认记忆 30 天。
- 会话只保存项目 ID、成员 ID、登录时间和过期时间，不保存明文密码。
- 会话保存在浏览器 LocalStorage。
- 成员被停用或会话过期时会要求重新登录。

安全说明：

- 这是本地协作权限，不是服务器级身份认证。
- 能直接修改项目文件的人仍可能绕过应用内限制，因此项目文件和修改包仍需通过备份、签名和团队流程保护。

## 8. 项目工作台

进入位置：登录后进入项目工作台。

左侧导航包含：

- 概览
- 文件
- 任务
- 术语
- 评论
- 统计
- 导入导出
- 设置

顶部显示项目面包屑、当前成员和退出登录按钮。

概览页显示：

- 综合、翻译、校对、审核进度。
- 审核关闭时显示“未启用审核”。
- 项目语言、文件数、任务数和词条状态统计。
- 修改包协作入口。

注意事项：

- 创建项目时的“启用任务”字段目前主要保存在工作流设置中，任务页仍会显示；当前版本没有按该字段隐藏整个任务模块。

## 9. 文件列表

进入位置：项目工作台 > “文件”。

可以执行：

- 查看项目文件和词条进度。
- 按文件分组筛选。
- 添加一个或多个源文件。
- 更新源文件。
- 导入单个文件的译文。
- 批量导入译文。
- 重命名项目中的显示名称。
- 设置逻辑分组。
- 隐藏、锁定或删除文件。
- 打开文件进入词条编辑。

权限不足时，相关操作会隐藏或禁用。隐藏文件不会进入成品导出；锁定文件会限制词条写入。

## 10. 添加源文件

进入位置：文件页 > “添加源文件”。

当前支持：

- `.txt`
- `.ks`
- `.json`
- `.jsonl`
- `.csv`

处理顺序：

1. 读取并解析源文件。
2. 写入 `source/`。
3. 写入 `entries/`。
4. 最后更新 `project.json`。

如果中途失败，`project.json` 不会新增坏记录；Textile 会尽量清理已经创建的源文件和词条目录，清理失败时会报告残留路径。

格式说明：

- TXT 和 KS 当前按非空文本行生成词条。
- JSON/JSONL/CSV 会按可识别的字段读取 key、speaker、source、target 等内容。
- 词条会按项目设置中的 `chunk_size` 写入 `chunk_0001.jsonl`、`chunk_0002.jsonl` 等分块；旧的单 chunk 项目仍可正常读取。

## 11. 更新源文件

进入位置：文件页 > 对应文件操作 > “更新源文件”。

更新时会：

- 重新解析源文件。
- 优先按词条 key 合并旧译文和审计信息。
- key 不匹配时再尝试按顺序位置合并。
- 更新源文件内容和词条文件。

注意事项：

- 大幅改变 key 或顺序可能导致旧译文无法正确匹配。
- 更新前应先导出 `.hproj` 备份。
- 当前更新流程不是完整事务；源文件或词条写入阶段出现异常时应检查备份和错误信息。

## 12. 导入译文

进入位置：文件页 > 对应文件操作 > “导入译文”，或使用批量导入。

匹配规则：

1. 优先按 key 匹配。
2. 没有 key 或 key 未命中时按 index 匹配。
3. 无法匹配的行计为跳过。

导入后：

- 已填入译文的未翻译词条会变为“已翻译”。
- `updated_by` 和 `translated_by` 记录当前执行导入的成员。
- 页面显示匹配和跳过数量。

批量导入时应确保文件名和项目文件关系明确；导入前建议备份。

## 13. 词条编辑页

进入位置：文件页点击文件，或从任务、评论跳转。

页面组成：

- 左侧词条列表。
- 中间词条编辑器。
- 右侧辅助面板：术语、评论、上下文、历史。

词条列表支持：

- 搜索 key、说话人、原文和译文。
- 按未翻译、已翻译、已校对、已审核、有争议筛选。
- 选择上一条、下一条。

编辑器支持：

- 查看词条 ID、key、说话人、原文、状态和审计信息。
- 编辑译文。
- 保存或保存并进入下一条。
- 执行校对、审核、退回、标记争议和解决争议。

注意事项：

- 隐藏、锁定或有争议的词条可能限制部分工作流操作。
- 保存者始终写入 `updated_by`，不会用任务负责人或旧值代替。

## 14. 翻译流程

典型流程：

1. 打开未翻译词条。
2. 填写译文。
3. 点击保存或翻译动作。
4. 状态变为“已翻译”。

审计字段：

- `updated_by`：本次保存成员。
- `updated_at`：本次保存时间。
- `translated_by`：译文从空变为非空或执行翻译动作的成员。

权限和限制：

- 需要 `entry.translate` 或相应编辑权限。
- 文件或词条锁定时不能正常翻译。
- 已进入后续流程的词条修改译文时，应结合团队规则重新校对和审核；当前状态函数在显式回到翻译状态时会清理后续审计信息。

## 15. 校对流程

项目可以设置 0 到 3 次校对。

操作：

1. 打开已翻译词条。
2. 检查和必要时修改译文。
3. 点击“校对通过”。
4. 系统增加 `proofread_count`，并把当前成员加入 `proofread_by`。

规则：

- 未达到所需次数时，词条显示“校对中 x/y”。
- 达到所需次数后状态进入“已校对”。
- 默认不允许译者校对自己的译文。
- 默认不允许同一成员重复完成多轮校对。
- 负责人可在设置中调整上述限制。
- 旧项目中字符串形式的 `proofread_by` 会按单元素数组读取。

## 16. 审核流程

审核开启时：

1. 词条必须满足项目要求的校对次数。
2. 有审核权限的成员点击“审核通过”。
3. 状态变为“已审核”。
4. `reviewed_by` 和 `updated_by` 记录当前审核成员。

默认不允许译者审核自己的译文，可在工作流设置中放开。

审核关闭时：

- 校对完成即作为主要流程完成。
- 统计页显示“未启用审核”。
- 综合进度忽略审核权重并重新归一化。
- 成品导出的“只导出流程完成词条”会改按校对完成过滤。

## 17. 争议标记

争议不是第五种词条状态，而是词条上的独立标记。

进入位置：

- 词条编辑器。
- 词条评论面板。

可以执行：

- 标记词条有争议。
- 填写争议原因。
- 需要时同时添加评论。
- 解决争议并记录处理人和时间。

有争议的词条仍保留原翻译状态，但工作流动作可能被限制。成品导出的争议报告会列出这些词条。

## 18. 上下文

进入位置：词条编辑页右侧 > “上下文”。

上下文用于记录：

- 场景说明。
- 前后文关系。
- 人物语气。
- 变量、标签或实现注意事项。

当前实现：

- 上下文主要保存在词条的 `context` 字段。
- 保存上下文会更新该词条的 `updated_by` 和 `updated_at`。
- 权限区分查看、新增、修改和删除。

注意事项：

- 修改包格式保留 `contexts/` 目录兼容能力，但普通词条上下文主要随 entries 一起导出。

## 19. 术语

进入位置：项目工作台 > “术语”，或词条编辑页右侧 > “术语”。

术语字段包括：

- 原词和推荐译名。
- 词性。
- 备注。
- 变体。
- 是否区分大小写。
- 创建人和更新时间。

术语页可以：

- 搜索、筛选和排序。
- 新建、编辑、删除。
- 导入 JSON、JSONL、CSV、XLSX。
- 下载示例 XLSX。
- 导出 JSONL。

词条辅助面板会根据当前原文匹配相关术语，也可以搜索全部术语。

当前限制：

- 不支持旧 `.xls` 文件。
- XLSX 导入读取第一张工作表。
- 删除术语前会二次确认。

## 20. 评论

进入位置：

- 项目工作台 > “评论”：浏览项目评论。
- 词条编辑页右侧 > “评论”：新增和处理当前词条评论。

支持：

- 新增评论。
- 回复评论。
- 标记解决、重新打开。
- 删除自己的评论；有权限者可删除任意评论。
- 从评论列表跳转到对应词条。
- 按文件和成员筛选。

评论是纯文本。删除父评论时会同时删除其回复。评论浏览页主要用于查找和跳转，具体编辑集中在词条评论面板。

## 21. 任务

进入位置：项目工作台 > “任务”。

任务类型：

- 翻译
- 校对
- 审核
- 术语
- 导出
- 自定义

任务状态：

- 未领取
- 已分配
- 进行中
- 已提交
- 已完成

任务可以关联：

- 文件。
- 词条范围。
- 明确的词条 ID 列表。
- 负责人、截止时间和校对轮次。

可用动作取决于权限和任务状态，包括创建、编辑、分配、领取、提交、完成、收回、重新打开和删除。

提交方式：

- 修改包提交。
- 由负责人处理。

旧数据中的 `git_hidden` 会显示为修改包提交，`git_manual` 会显示为由负责人处理，不会继续提供 Git 选项。

任务进度使用与项目统计一致的工作流和权重。删除任务前会二次确认。

## 22. 统计

进入位置：项目工作台 > “统计”。

显示内容：

- 综合进度。
- 翻译进度。
- 校对进度。
- 审核进度或“未启用审核”。
- 总词条数。
- 未翻译、已翻译、已校对、已审核、有争议数量。
- 当前实际归一化后的权重。

统计口径：

- 翻译进度：已经进入翻译及后续状态的词条比例。
- 校对进度：达到项目要求校对次数的词条比例。
- 审核进度：审核开启时已审核词条比例。
- 综合进度：三个阶段按项目权重计算。
- 审核关闭时审核权重置 0，其余权重重新归一化。

当前统计按词条实时读取，不依赖单独统计数据库。

## 23. 修改包导出

进入位置：项目工作台 > “导入导出”。

导出模式：

### 我的全部修改

导出当前成员修改的词条、评论、创建的术语、相关任务和事件。词条筛选依赖可靠的 `updated_by`。

### 所选任务修改

只导出选定任务范围内、由当前成员修改的词条和相关内容。

### 项目维护修改

用于有权限的维护人员导出术语、上下文、任务、项目配置、成员和日志等维护数据。可能包含成员凭据变更，普通成员无权使用。

### 签名项目更新包

由有发布权限的负责人生成权威项目快照，包含：

- 全部词条、评论、术语、任务。
- 源文件。
- 项目配置。
- 成员公开信息。
- 日志。

项目更新包必须签名。它不会发布成员密码哈希、密码盐或私钥。

签名说明：

- 普通修改包在当前成员已加载私钥时可以签名；没有私钥时仍可导出未签名普通包。
- 项目更新包必须有有效私钥和签名。

## 24. 修改包导入

进入位置：项目工作台 > “导入导出” > “导入修改包”。

权限：

- 普通成员通常不能合并其他成员的普通修改包。
- 负责人、管理员或配置了相应权限的成员负责导入普通修改包。
- 所有拥有接收项目更新权限的成员可以接收签名项目更新包，默认只读成员也包括该权限。

导入前会检查：

- `manifest.json` 是否存在。
- schema 是否支持。
- 项目 ID 是否匹配。
- 内容哈希是否通过。
- 签名状态。
- 修改包类型和风险。
- 目标数据路径是否可访问。
- 冲突列表是否已生成。

预检查失败时不会开始写入项目文件。

维护修改包如果涉及项目设置、成员、权限或凭据，需要额外确认。负责人账号凭据或 owner 权限变化还需要再次确认。

## 25. 项目更新包

项目更新包用于负责人向普通成员分发统一项目进度。

接收条件：

- 项目 ID 相同。
- 内容完整性通过。
- 签名有效。
- 签名人当前拥有发布项目更新的权限。
- `base_revision` 与本地当前 revision 相同。
- 包内 `target_revision` 与项目配置一致。
- 当前成员没有未导出的个人修改。

接收后：

- 项目正文、源文件、术语、任务、公开成员信息和日志更新。
- 本地成员密码哈希和盐被保留。
- 私钥从不由项目更新包分发。

如果更新基线不匹配，通常说明漏接了前一个更新包或本地版本不同，需要从负责人获取正确的连续更新或新的 `.hproj` 基线。

## 26. 冲突处理

进入位置：导入普通修改包后，如果发现冲突，页面显示“冲突处理”。

当前冲突主要比较同一词条的译文和状态。每条冲突可以选择：

- 保留主项目。
- 使用修改包。
- 手动处理。
- 跳过。

手动处理允许填写合并后的译文和状态。所有冲突必须有处理结果才能应用修改包。

当前限制：

- 项目更新包是权威快照，不进入逐词条冲突界面，而通过 revision、签名和未导出修改检查保护。
- 普通导入具备预检查，但不是完整事务系统。导入大型或维护修改包前必须先做 `.hproj` 备份。

## 27. 成品导出

进入位置：项目工作台 > “导入导出” > “导出成品”。

支持格式：

- JSON
- TXT 对照
- CSV
- KS

可选项：

- 只导出流程完成词条。
- 包含原文。
- 包含键值。
- 生成报告。
- 生成 manifest；该选项在设置页保存，导入导出页使用项目默认值。

成品 ZIP 包可能包含：

```text
release/
manifest.json
reports/untranslated.txt
reports/disputes.txt
reports/term-check.txt
```

隐藏文件和隐藏词条不会进入成品。导出结果由浏览器下载，不会自动写入项目的 `exports/` 目录。

## 28. 项目备份与 `.hproj`

进入位置：

- 导入导出页 > “导出 Textile 项目备份”。
- 设置 > “协作与备份”。

当前 `.hproj` 包含存在的以下内容：

```text
project.json
members.json
source/
entries/
terms/
tasks/
comments/
logs/
exports/
changes/
```

不存在的可选目录会跳过。`source/` 存在时一定会打包。

建议：

- 添加、更新或删除源文件前备份。
- 导入维护修改包或大量成员修改前备份。
- 转让 owner、调整权限和重置密码前备份。
- 从 `.hproj` 导入后确认本地项目文件夹可正常打开，不要把唯一 `.hproj` 当作工作目录。

## 29. 成员管理

进入位置：设置 > “成员管理”。

可执行操作取决于权限：

- 新增成员。
- 分配一个或多个角色。
- 修改成员角色。
- 停用成员。
- 重置成员密码。
- 修改自己的密码。
- 转让 owner。

约束：

- 项目只能有一个 owner。
- owner 不能被普通停用流程移除。
- 转让 owner 需要两次确认，原 owner 会变为管理员。
- 系统会防止保存导致没有任何可用管理成员的权限配置。

密码更新只写入当前项目的 `members.json`。常规项目更新包不会传播密码变更；需要跨副本同步成员凭据时，应由负责人通过受控维护流程或重新分发基线项目，不能依赖常规项目更新包。

## 30. 角色与权限

进入位置：设置 > “角色与权限”。

内置角色：

- 项目负责人 owner
- 管理员 admin
- 技术负责人 tech_lead
- 翻译 translator
- 校对 proofreader
- 审核 reviewer
- 发布负责人 publisher
- 术语负责人 term_manager
- 只读成员 readonly

权限按 action 管理，涵盖项目、成员、文件、词条、术语、上下文、任务、评论、修改包、成品、密钥和维护。

有效权限计算：

1. 合并成员所有角色的权限。
2. 加入成员 `allow_permissions`。
3. 移除成员 `deny_permissions`。

`deny_permissions` 优先级最高。owner 的项目、成员、角色和备份等关键权限会锁定。

## 31. 工作流设置

进入位置：设置 > “工作流”。

可以设置：

- 校对次数 0 到 3。
- 是否需要审核。
- 是否允许译者校对自己的译文。
- 是否允许译者审核自己的译文。
- 是否允许同一成员完成多轮校对。

这些设置会影响：

- 词条按钮和状态。
- 校对次数判断。
- 项目、文件、任务和统计进度。
- 完成状态判断。

修改工作流前应先确认现有词条审计字段是否仍符合团队规则。

## 32. 进度权重

进入位置：设置 > “进度权重”。

默认值：

- 翻译 40%
- 校对 30%
- 审核 30%

保存时总和必须等于 100。审核关闭后，统计计算会忽略审核权重并重新归一化翻译和校对权重。

如果校对次数设置为 0，建议手动把校对权重设为 0；当前统计页会提示，但不会自动修改项目配置。

## 33. 签名密钥

进入位置：设置 > “我的身份密钥”。

用途：

- 给修改包签名。
- 验证修改包来源和内容完整性。
- 项目更新包必须使用有效负责人密钥签名。

密钥组成：

- 公钥和 key ID 保存在 `members.json`。
- 私钥只保存在当前运行内存中。
- 用户可以显式导出 `member-key.json`，以后再导入。

重要注意事项：

- 重新加载页面或关闭应用后，内存中的私钥会消失。
- 需要签名时必须重新导入私钥文件，除非本次运行刚生成。
- `member-key.json` 当前包含未加密的私钥 JSON，必须像密码一样保管。
- 不要把私钥文件放进项目文件夹、`.hproj`、修改包、聊天群公共文件或代码仓库。
- 生成新密钥、撤销密钥和撤销其他成员公钥都有确认提示。
- 成员身份密钥与 Tauri 程序发布签名密钥完全不同。

## 34. 程序更新

进入位置：设置 > “关于 Textile / 更新”，或页面底部更新提示。

Web/PWA：

- 从 `/version.json` 检查版本。
- PWA 可以在后台准备新资源。
- 安全页面可刷新到新版；正在编辑、导入导出或执行写操作时会暂缓。
- 如果 `download_url` 为空，按钮显示“未配置发布地址”并禁用。

Tauri 桌面版：

- 使用 Tauri Updater 检查、下载、验签、安装和重启。
- 正在执行项目写入或处于编辑页面时会等待安全时机。

当前限制：

- `src-tauri/tauri.conf.json` 的 updater 公钥和 endpoints 目前为空，正式桌面自动更新尚未配置完成。
- `public/version.json` 的下载地址目前为空。
- stable/beta 通道在 UI 中可选择，但实际可用版本取决于发布清单。

## 35. 危险操作

进入位置：设置 > “危险操作”。

### 重建统计

当前统计按词条实时计算。“重建统计”主要刷新项目摘要，不删除数据。

### 清理缓存

可以选择清理：

- 导入预览缓存。
- 导出临时缓存。
- 更新检查缓存。
- 最近项目记录。
- 当前登录会话。

不会删除项目正文、成员数据或项目文件。私钥本来只在内存中，清理缓存不代表安全备份私钥。

### 从启动页移除项目

当前语义只有：

- 从最近项目移除。
- 清除当前项目会话。
- 返回项目启动页。
- 磁盘文件不会被删除。

Textile 当前不会递归删除磁盘项目文件夹。如果要彻底删除，请先确认备份，再在操作系统中手动删除项目目录。

### 文件、术语、任务和密钥操作

删除文件、删除术语、删除任务、撤销密钥、停用成员、转让 owner 等都有确认提示。确认前应检查备份和影响范围。

## 36. 常见问题

### 为什么最近项目打不开？

浏览器可能丢失了文件夹授权。重新使用“打开 Textile 项目文件夹”选择同一目录。

### 为什么原 `.hproj` 没有随着本地编辑变化？

`.hproj` 是备份和分发包。导入后编辑的是新生成的本地项目文件夹，不会原地修改原 `.hproj`。需要分发或备份时，请重新导出新的 Textile 项目备份。

### 为什么普通成员不能导入其他人的普通修改包？

普通修改包会合并内容，默认由负责人或有合并权限的成员处理。普通成员通过签名项目更新包接收统一进度。

### 项目更新包会更新密码或私钥吗？

不会。项目更新包只带成员公开信息，并保留接收方本地密码哈希和盐。私钥从不进入项目更新包。

### 为什么不能接收项目更新？

常见原因：

- 当前有未导出的个人修改。
- 项目 ID 不匹配。
- 更新包签名无效。
- 基线 revision 不匹配。
- 更新包不是当前允许发布者签名。

### 为什么不能发布项目更新包？

项目更新包必须由有发布权限的成员签名。请确认当前成员拥有发布权限，已生成公钥，并在本次运行中加载了对应私钥。

### 为什么“只导出流程完成词条”没有内容？

该选项会按当前项目工作流过滤。审核开启时需要已审核；审核关闭且校对开启时需要完成校对；审核和校对都关闭时需要有译文。

### 为什么任务功能关闭后任务页还在？

当前 `enable_tasks` 设置尚未控制任务页导航，任务模块仍可见。这是当前版本限制。

### 为什么源文件更新后部分译文没有保留？

更新优先按 key、其次按 index 合并。源文件 key 和顺序同时发生大幅变化时，旧译文可能无法匹配。更新前应先备份。

### 项目权限能防止直接改文件吗？

不能完全防止。权限主要约束 Textile 内的协作操作；本地文件持有者仍可直接修改文件。重要交付应结合备份、修改包签名、验签和团队流程。

## 37. 团队使用建议

- 负责人保留一份主项目文件夹，并定期导出 `.hproj`。
- 每轮工作前先分发签名项目更新包。
- 成员接收更新后再开始编辑，减少基线分叉。
- 成员按任务或工作批次导出修改包，不要长期积累未提交修改。
- 负责人每次导入前先预览内容、检查签名和冲突。
- 大批量导入、源文件更新、成员权限调整前先备份。
- 私钥单独安全保存，不放进项目和修改包。
- 成品导出前检查未翻译、争议和术语报告。
